# Описание

**Задача.** Трекер одновременно отслеживает несколько штрихкодов в кадре. Каждая физическая метка получает свой идентификатор траектории **`track_id`**, между кадрами он сохраняется; текст по траектории сглаживают. За связь между кадрами отвечает библиотека **Norfair**, которая сопоставляет центры зон от кадра к кадру.

**На одном кадре (перед передачей в Norfair).** Если центры двух распознанных штрихкодов находятся ближе чем на *150 пикселей*, они считаются одной физической меткой и объединяются. Можно включить режим «сначала по строке» — тогда склеиваются только штрихкоды с одинаковым распознанным текстом. Это нужно, чтобы разные коды, стоящие рядом, не слипались просто из-за близости на картинке.

**Между кадрами.** Norfair сопоставляет центры зон с предсказанными позициями, используя евклидово расстояние. Порог сопоставления по умолчанию равен *1.2* от порога склейки — то есть при базовом пороге *150 пикселей* получается около *180 пикселей*. Сколько кадров объект может отсутствовать, прежде чем трекер его забудет, определяется внутренними правилами Norfair (`hit_counter_max`). Обычно это значение не меньше *4*, а с учётом пропусков (порядка *5*) и запаса получается около *7 кадров*.

**Текст.** По каждой траектории строка сглаживается модой по сохранённой истории Norfair; длина буфера задаётся параметром окна (по умолчанию порядка *7*).

**Масштабирование.** Все пороги заданы в пикселях. При смене разрешения видео эти значения нужно пересчитывать пропорционально.

**Примечание про статистику.** Если грубо считать каждый кадр независимой попыткой распознать штрихкод с вероятностью успеха *p*, и мы хотим, чтобы вероятность серии из нескольких ошибок подряд была не больше *ε = 0.001*, то необходимое число попыток оценивается как *ln(ε) / ln(1-p)*. На видео соседние кадры очень похожи, поэтому попытки нельзя считать независимыми — требуется дополнительный множитель *k > 1*.
